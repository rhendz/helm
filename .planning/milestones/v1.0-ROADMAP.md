# Milestone v1.0: Helm Orchestration Kernel v1

**Status:** SHIPPED 2026-03-14
**Phases:** 1-4
**Total Plans:** 14

## Overview

`v1.0` established Helm’s reusable orchestration kernel. The milestone shipped durable workflow persistence, typed specialist dispatch, approval and revision semantics, adapter-gated sync execution with recovery and replay, and one representative weekly scheduling workflow that proves the kernel end to end through approved side effects.

## Phases

### Phase 1: Durable Workflow Foundation

**Goal**: Helm has a durable workflow run model with persisted step state, artifacts, validation results, and inspectable run status.
**Depends on**: Nothing
**Plans**: 3 plans

Plans:

- [x] 01-01: Add workflow run, step state, and artifact persistence models plus repositories.
- [x] 01-02: Add typed workflow schemas, validation boundaries, and validation-failure handling.
- [x] 01-03: Expose run status and artifact inspection through API and Telegram-friendly read paths.

### Phase 2: Specialist Dispatch And Approval Semantics

**Goal**: Helm can execute typed `TaskAgent` and `CalendarAgent` steps, create approval checkpoints, support revision versioning, and resume from decisions safely.
**Depends on**: Phase 1
**Plans**: 3 plans

Plans:

- [x] 02-01: Implement specialist step execution contracts and invocation recording inside the orchestration kernel.
- [x] 02-02: Implement approval request storage, decision handling, and pause/resume transitions.
- [x] 02-03: Add proposal revision/versioning flow for approval-driven rework.

### Phase 3: Adapter Writes And Recovery Guarantees

**Goal**: Approved workflows can write through adapters safely with strong idempotency, retry, replay, and sync lineage guarantees.
**Depends on**: Phase 2
**Plans**: 5 plans

Plans:

- [x] 03-01: Implement task-system and calendar adapter contracts with sync-record persistence.
- [x] 03-02: Add idempotency keys, duplicate-write prevention, and reconciliation behavior across retry and resume.
- [x] 03-03: Add retry vs replay semantics, recovery state tracking, and operator-safe failure handling.
- [x] 03-04: Expose recovery semantics through the shared workflow status projection.
- [x] 03-05: Wire replay and recovery entrypoints through API, worker, and Telegram surfaces.

### Phase 4: Representative Scheduling Workflow

**Goal**: Helm proves the kernel with the weekly scheduling workflow from raw user request through approved downstream writes.
**Depends on**: Phase 3
**Plans**: 3 plans

Plans:

- [x] 04-01: Implement the fixed weekly scheduling workflow on top of the kernel primitives.
- [x] 04-02: Add end-to-end verification, restart/recovery checks, and operator-facing completion summaries for the representative flow.
- [x] 04-03: Fix replay-recovery status precedence for completed representative runs and add focused replay/status/Telegram regressions.

## Milestone Summary

**Key Decisions:**

- Keep workflow durability in dedicated `workflow_*` storage and typed orchestration schemas rather than extending legacy domain-specific persistence.
- Make approval and revision semantics kernel-owned and expose them through one shared status projection for API and Telegram.
- Treat sync execution, recovery, and replay as durable lineage concerns owned by the kernel rather than by app-layer retries.
- Use one representative weekly scheduling workflow to prove the kernel before expanding workflow breadth.

**Issues Resolved:**

- Closed the replay/final-summary gap so durable final-summary lineage refreshes when replay adds new downstream sync state.
- Added missing phase verification artifacts for phases 1-3 so milestone audit can cross-reference requirements, summaries, and verification evidence.
- Backfilled compliant Nyquist validation records across all four phases.

**Issues Deferred:**

- Cross-phase integration review remained manual in this session because the dedicated integration-checker role was unavailable.
- Next-milestone workflow expansion and richer operator comparison flows remain deferred beyond v1.0.

**Technical Debt Incurred:**

- The worktree still contains unrelated historical planning files and `.planning/config.json` changes that should be handled deliberately during future planning cleanup.
